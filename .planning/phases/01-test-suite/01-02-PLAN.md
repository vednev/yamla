**Section 2: utc_to_epoch tests**

1. **"utc_to_epoch Unix epoch"** — `utc_to_epoch(1970, 1, 1, 0, 0, 0)` must equal `0`.

2. **"utc_to_epoch known date 2024-01-15 10:30:45"** — Must equal `1705312245`. Verify with: `days = 19737`, `seconds = 19737*86400 + 10*3600 + 30*60 + 45 = 1705312245`.

3. **"utc_to_epoch leap year 2024-02-29"** — `utc_to_epoch(2024, 2, 29, 0, 0, 0)` must be valid (2024 is a leap year). The value should be `1709164800`.

4. **"utc_to_epoch end of day"** — `utc_to_epoch(2024, 1, 1, 23, 59, 59)` — verify it's `utc_to_epoch(2024, 1, 1, 0, 0, 0) + 86399`.

5. **"utc_to_epoch year 2000"** — `utc_to_epoch(2000, 1, 1, 0, 0, 0)` must equal `946684800`.

**Section 3: parse_timestamp tests**

6. **"parse_timestamp valid ISO 8601"** — `parse_timestamp("2024-01-15T10:30:45.123+00:00")` must equal `1705312245123` (epoch_s * 1000 + 123ms).

7. **"parse_timestamp midnight"** — `parse_timestamp("2024-01-01T00:00:00.000+00:00")` must equal `1704067200000`.

8. **"parse_timestamp max milliseconds"** — `parse_timestamp("2024-01-15T10:30:45.999+00:00")` — ms component is 999.

9. **"parse_timestamp too short"** — `parse_timestamp("2024-01-15T10:30")` must return `-1` (string < 23 chars).

10. **"parse_timestamp bad separator"** — `parse_timestamp("2024/01/15T10:30:45.123+00:00")` must return `-1` (slash instead of dash at position 4).

11. **"parse_timestamp empty"** — `parse_timestamp("")` must return `-1`.
